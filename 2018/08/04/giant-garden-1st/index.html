【巨人花园】第1回：序言

前言

今天开拓了一个新的系列，称为“巨人花园”。为什么我要称这个版块为“巨人花园”？因为这系列的记录都是基于国内外大神们的结晶上的，也就是记录了通过他们写的一些文章、书籍或教程，我学习到了什么？我学习的感受和心得等等。
初次有这样的念头是因为看到张偶老师在segmentfault发布的每日前端系列，因为是代码所以必须要自己实践才行，其中也有一些不太熟悉的CSS属性和技巧，也很适合自己去深挖。随后又在国外看到了很多这样有趣的文章，所以我按捺不住，终于开设了这个板块。
之所以很想强调出“巨人”的概念，是因为我发现有很多人对他人的文章进行搬运、翻译或者二次修改后都不会很清楚、明显的标注原作，反而还有意“突出自己的形象”，我认为这是一种很不好的行为，所以我带着一些小崇拜，稍微能帮原作者宣传，也希望自己能持续学习，而记下这些我的这些心得。

效果分析

先从静态的结构来分析，它实际上是有3个元素：

  • 作为背景的浅蓝色圈
  • 深蓝色圆弧
  • 表示进度的文字

从动态效果来分析，我们要改变的有：

  • 深蓝色圆弧的跨越度
  • 进度百分号数字

效果实现

先把东西都准备好

我们要在canvas上作画，所以如果还是canvas小白，那么请先在HTML结构中加入一个canvas标签：

<canvas id="canvas" width='500' height="500">
   <p>你的浏览器不支持Canvas</p>
</canvas>

假如你的浏览器不支持canvas，标签中间的内容才会出现QAQ。

然后我们在JS中获取canvas元素，随后从这个canvas元素里获取2d画布！我们是在画布上画画哦！

var canvas = document.getElementById('canvas'), 
context = canvas.getContext('2d’);

Q：萌新提问啦，我知道我画的是平面图，所以这里传了一个2d进去，那我传3d进去是不是就可以画3d图了？
A：小年糕帮你搜索了一下，官方API是说～在未来，如果 canvas 标签扩展到支持 3D 绘图，getContext() 方法可能允许传递一个 “3d” 字符串参数，所以现在还都只是传2d的。
参考文章：https://segmentfault.com/a/1190000008871433

画圆弧

context.arc(圆心x坐标,圆心y坐标,圆半径,起始角（弧度计算）,结束角,true = 逆时针/false = 顺时针);
更多参考：http://www.w3school.com.cn/tags/canvas_arc.asp

因为我们要在画布中心画这个圆，所以要计算出它的圆心。随后开始画这个圆弧，起点是在圆的上方，所以这里是-Math.PI/2为起始位置，这里的n是一个动态值，我们就根据这个来让圆弧慢慢增长。

centerX = canvas.width/2;
centerY = canvas.height/2;
context.arc(centerX, centerY, 100 , -Math.PI/2, -Math.PI/2 + n*rad, false);
rad = Math.PI*2 / 100;
speed = 0.1;   //变化速度

类似的，把两个圆弧都画出来。

【补充】ctx.save()和ctx.restore()

save：用来保存Canvas的状态。save之后，可以调用Canvas的平移、放缩、旋转、错切、裁剪等操作。
restore：用来恢复Canvas之前保存的状态。防止save后对Canvas执行的操作对后续的绘制有影响。

完成动画

requestAnimationFrame函数来调用这个你要循环（用于刷新帧）的函数，浏览器刷新频率决定函数调用的频率：

(function drawFrame() {
   window.requestAnimationFrame(drawFrame, canvas);
    ctx.clearRect(0, 0, canvas.width, canvas.height);    
         whiteCircle();
         text(speed);
         blueCircle(speed);     
         if(speed > 100) speed = 0;
         speed += 0.1; // 更改speed 下次进行绘制的时候就变了
  })()
}

Q：requestAnimationFrame和setTime系列方法有什么不同吗？
A：好，小年糕继续帮你搜索，结果如下。
我们需要使用JS来编写动画效果，那么时间的把握其实是比较重要的。大多数显示器的刷新频率是60Hz，大概相当于每秒钟重绘60次。浏览器一般会对重绘操作加以限制，让它们不超过显示器的重绘频率，因此，最平滑动画的最佳循环间隔是1000ms/60，约等于16.6ms。
setTimeout和setInterval的问题是，它们都不精确。它们的内在运行机制决定了时间间隔参数实际上只是指定了把动画代码添加到浏览器UI线程队列中以等待执行的时间。如果队列前面已经加入了其他任务，那动画代码就要等前面的任务完成后再执行。
requestAnimationFrame采用系统时间间隔，保持最佳绘制效率，不会因为间隔时间过短，造成过度绘制，增加开销；也不会因为间隔时间太长，使用动画卡顿不流畅，让各种网页动画效果能够有一个统一的刷新机制，从而节省系统资源，提高系统性能，改善视觉效果。但是这个API也是要注意一下兼容问题的。
更多参考：https://www.cnblogs.com/xiaohuochai/p/5777186.html
